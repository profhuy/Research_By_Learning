\section{Methodology}\label{sec:method}

This section describes the experimental design, the dataset, the exact
LLM configuration, and the execution pipeline used to answer our
research question. The protocol was fixed in the approved proposal
before any data was inspected. Every deviation from that protocol is
reported explicitly in Sections~\ref{sec:method:prompts}
and~\ref{sec:method:repro}.

% ------------------------------------------------------------------
\subsection{Study Design}
\label{sec:method:design}

We conduct an observational agreement study: an LLM and human
annotators independently label the same bug reports along three
quality dimensions---Observed Behavior (OB), Expected Behavior (EB),
and Steps to Reproduce (S2R)---and we quantify how closely the LLM
reproduces the annotator consensus. The LLM is therefore treated as a
candidate \emph{quality gate} placed in front of developer triage,
not as a text generator.

The study has two phases, and we keep them separate because they
answer different questions:

\begin{itemize}
  \item \textbf{Phase~1 --- agreement ($N=30$).} Every report in this
        subset carries a human consensus label, so this is the phase
        that measures how reliably the model matches human judgment.
        All agreement results reported in Section~\ref{sec:results}
        come from this phase.
  \item \textbf{Phase~2 --- scale-up ($N=250$).} The complete corpus
        was labeled by the model under production-like conditions.
        Human annotation of this corpus was still in progress at the
        time of writing, so Phase~2 supports claims about
        \emph{operational feasibility}---throughput, cost, output
        validity, and the distribution of labels the gate would
        emit---but no agreement claim. We state this limit explicitly
        rather than substituting a proxy for the missing labels.
\end{itemize}

Both phases share one pipeline: reports are mined and stored verbatim;
each report is sent to the model in a single stateless request that
must return a strict JSON object; responses are parsed and any empty or
malformed response is marked \texttt{INVALID} and never filled in by
hand; and, where human labels exist, Cohen's Kappa is computed between
the two annotators and between the model and the annotator consensus.

% ------------------------------------------------------------------
\subsection{Subject Systems and Dataset}
\label{sec:method:dataset}

We selected three widely used open-source projects that differ in
domain and in reporting culture: \texttt{pandas} (data analysis
library), \texttt{scikit-learn} (machine-learning library), and
\texttt{vscode} (a large IDE with a very high issue volume). This
choice lets us observe whether the gate behaves consistently across a
library-style and an application-style project.

Table~\ref{tab:dataset} reports the composition of both phases. The
corpus contains 250 bug reports created between February 2024 and June
2026; both open and closed issues are included (156 closed, 94 open),
because closing status is not part of the construct we measure. The
annotated subset ($N=30$) is a strict subset of the corpus.

Issues were sampled at random from each project's issue tracker.
The sampling seed was not recorded, so the exact sample cannot be
regenerated by a third party; the sampled reports themselves are
committed to the repository so that our specific results remain
verifiable. We report this as a limitation in
Section~\ref{sec:threats} rather than presenting the sampling as
seed-reproducible.

Only the issue \emph{title} and \emph{body} are given to the model and
to the annotators. Comments, linked pages, commit history, and the
eventual fix are excluded, since a quality gate must decide at
submission time, when none of that information exists yet. Report
bodies average 3{,}668 characters (median 2{,}760).

\subsection{Annotation Rubric and Ground Truth}
\label{sec:method:groundtruth}

Each of OB, EB, and S2R receives exactly one of four labels:
\texttt{Sufficient} (present and clear enough to act on),
\texttt{Ambiguous} (present but vague), \texttt{Missing} (not stated),
or \texttt{Incorrect} (stated but contradictory or unusable). The three
dimensions are judged independently, and annotators are explicitly
forbidden to reward domain knowledge that is not written in the report
text---the same restriction the model operates under, which keeps the
two conditions comparable. The resolved labels are referred to
throughout this paper as the \emph{annotator consensus}.

Two annotators labeled the Phase~1 subset. Annotator~1 labeled all 30
reports; annotator~2 independently labeled 10 of 30 (33\%) for
inter-annotator agreement, and disagreements were to be resolved by
discussion into the consensus columns. Annotation of the remaining 220
reports of the corpus is ongoing and is not used anywhere in this
paper.

\subsection{LLM Configuration}
\label{sec:method:llm}

Table~\ref{tab:config} lists the configuration. We use GPT-4o~mini,
which is a deliberate choice rather than a convenience one: a quality
gate would run on every incoming issue, so cost per report is a design
constraint, and a small model is the realistic deployment candidate.
The same model family has been used in recent bug-report work by
Akyol~et~al.~\cite{akyol2026improbr}.

We set \texttt{temperature}~$=0.0$ to make the labeling as close to
deterministic as the API allows. This differs from generation-oriented
studies, which use sampling to obtain diverse candidates
(e.g., $t=0.7$ in LIBRO~\cite{kang2024libro}); for a classification
gate, output stability is preferable to diversity. Each report is one
independent, stateless request: no conversation history, no examples in
the prompt (zero-shot), and no self-consistency voting.

The system prompt defines OB, EB, and S2R, defines the four labels,
forbids the use of outside knowledge and of inference over absent
information, and requires the reply to be strictly the JSON object
\texttt{\{"OB":\{"label","reason"\},"EB":\{...\},"S2R":\{...\}\}} with
no markdown and no commentary. The user message contains only the issue
title and body. Requiring a machine-parsable JSON schema follows the
prompt-design practice reported by Bo~et~al.~\cite{bo2024chatbr} and
Acharya and Ginde~\cite{acharya2025enhance}, and it is what makes an
\texttt{INVALID} response detectable rather than silently
mis-interpreted.

\subsection{Prompt Versions}
\label{sec:method:prompts}

Phase~1 used prompt \texttt{v1.0}. Its per-dimension agreement was high
for OB but weak for S2R, and manual inspection of the disagreements
showed a systematic cause: \texttt{v1.0} led the model to expect
explicit, enumerated steps, so reports whose reproduction path was a
runnable code snippet or a single command---the dominant style in
\texttt{pandas} and \texttt{scikit-learn}---were labeled
\texttt{Missing}, whereas the annotators accepted them as actionable.

Phase~2 therefore used \texttt{v3.0-candidate-s2r-relaxed}, which
changes two things relative to \texttt{v1.0}:

\begin{itemize}
  \item \textbf{S2R relaxed:} a concrete code snippet, command block, or
        short workflow that identifies the triggering action counts as
        \texttt{Sufficient} even without numbered steps; \texttt{Missing}
        is reserved for reports with no actionable reproduction path at all.
  \item \textbf{EB tightened:} EB is \texttt{Sufficient} only when the
        report explicitly states what should have happened; an
        expectation that is merely guessable from context is not enough.
\end{itemize}

Two honest consequences follow, and we state both. First, the change is
a deviation from the frozen-prompt protocol; it aligns the prompt with
the annotation rubric and does not touch the rubric, the metric, the
threshold, or the model. Second---and more importantly for how our
results should be read---\texttt{v3.0} has \emph{not} been shown to fix
the S2R weakness. It was applied to the corpus that has no human labels
yet, so its effect on agreement is untested. The agreement numbers in
Section~\ref{sec:results} characterise \texttt{v1.0}, and evaluating
\texttt{v3.0} against human labels remains future work.

\subsection{Execution Pipeline and Error Handling}
\label{sec:method:pipeline}

Because an API-based pipeline can fail in ways that quietly corrupt a
dataset, error handling was defined before the run rather than
improvised during it:

\begin{itemize}
  \item \textbf{Rate limiting.} Requests are retried up to five times
        with exponential backoff and jitter ($2^{n}$~seconds plus a
        random offset). The jitter avoids synchronised retries; the cap
        avoids masking a persistent outage as slowness.
  \item \textbf{Other errors.} Any non-rate-limit exception is raised
        instead of retried, so that a genuine defect surfaces
        immediately rather than being retried into silence.
  \item \textbf{Empty or unparsable responses.} The row is marked
        \texttt{INVALID} in \texttt{raw\_json\_status}. Labels are never
        filled in by hand and invalid rows are never deleted; they are
        counted separately so that the invalid rate is reported next to
        the metric.
  \item \textbf{Logging.} Every call appends one line to the API log
        recording timestamp, repository, issue id, the model string
        actually returned, success/failure, parse status, token usage,
        cost, and any error message. The returned model string is logged
        (not assumed) so that a silent model rollover is detectable.
\end{itemize}

In practice, all 30 Phase~1 calls and all 250 Phase~2 calls succeeded
and returned parsable JSON: 0/250 \texttt{INVALID}, no retries
triggered, and no format change during the run. One anomaly was
observed: 2 of the 250 responses reported the model alias
\texttt{gpt-4o-mini} instead of the pinned snapshot
\texttt{gpt-4o-mini-2024-07-18}; both parsed correctly and were kept,
and we note them because exact snapshot pinning is a reproducibility
concern for API-based studies.

\subsection{Measurement Procedure}
\label{sec:method:metrics}

Agreement is measured with Cohen's Kappa~\cite{cohen1960}, which
corrects for the agreement expected by chance, and we interpret
magnitudes following Landis and Koch~\cite{landis1977}. Kappa is also
the measure used by recent bug-report studies that compare human
raters, which makes our numbers comparable to
theirs~\cite{mahmud2025astrobr,akyol2026improbr}. We compute it
(i)~between the two annotators, to establish that the rubric itself is
reliable, and (ii)~between the model and the annotator consensus, which
is the quantity the research question asks about. Both are computed
overall and per dimension, since a single overall value can hide one
weak dimension. The pass threshold fixed in the proposal is
$\kappa \geq 0.70$.

Two properties of our label distribution require care, and we handle
them explicitly rather than reporting a number that looks better than
the data supports.

First, the distribution is heavily skewed toward \texttt{Sufficient}.
Raw agreement therefore overstates the gate's ability, and we report it
only alongside Kappa, never instead of it.

Second, the skew is total on the OB dimension: every report in the
annotated subset is labeled \texttt{Sufficient} for OB by both the
model and the annotators. When two raters use a single category,
chance-expected agreement is $1$ and Kappa evaluates to $0/0$; it is
\emph{undefined}, not $1$. We therefore report OB as raw agreement with
$N$ and an explicit note that Kappa is undefined, and we exclude OB
from any averaged Kappa. Reporting an undefined coefficient as perfect
agreement would invert the finding: a dimension on which the data
carries no discriminative information would appear to be the gate's
strongest result.

\subsection{Cost, Runtime, and Reproducibility}
\label{sec:method:repro}

Phase~1 issued 30 API calls on 2026-06-29, consuming 50{,}174 prompt
tokens and 3{,}545 completion tokens for approximately US\$0.010 in
under two minutes. Phase~2 issued 250 API calls on 2026-07-06,
consuming 449{,}688 prompt tokens and 30{,}894 completion tokens for
approximately US\$0.086 in about 15 minutes, i.e.\ roughly US\$0.0003
per bug report. This is the main positive result of Phase~2: screening
an entire issue backlog with this gate costs cents and runs in minutes,
so the binding constraint on deployment is accuracy, not cost.

The artifacts needed to check our results are in the project
repository: the sampled corpus, the human labels for the annotated
subset, the frozen prompts, the parsed model outputs, and the per-call
API logs. As noted in Section~\ref{sec:method:dataset}, the sampling
seed was not recorded, so the sample can be inspected but not
regenerated. No API key is stored in the repository; credentials are
read from the environment at runtime.
